# Project Context Updater

这个 skill 用于检测并更新项目的上下文索引。当用户提到新的文档链接、路径或约束时，检查这些上下文是否已被索引，并询问用户是否需要添加。

## 工作流程

### 步骤 1：读取上下文索引

当这个 skill 被调用时，首先读取 `AGENTS.md` 文件，查看当前项目的上下文索引文件路径，然后读取这些索引文件的内容：
- `.agents/remote-docs-index.md`（远程文档索引）
- `.agents/project-docs-index.md`（项目文档索引）
- `.agents/constraints.md`（编码约束）

### 步骤 2：检测新的上下文

分析用户的提示词，检测用户是否提供了新的上下文。检查以下三种情况：

#### 情况 A：新的远程文档上下文

**触发条件：** 用户提供了远程文档链接

**示例：**
```
用户："请你根据 https://example.com/docs/ 文档，实现项目内的功能 A"
```

**判断逻辑：** 检查 `.agents/remote-docs-index.md` 中是否已存在该远程文档或域名

**如果不存在：** 向用户提问：
> 是否要将这个新的文档添加到 `.agents/remote-docs-index.md` 索引文件中？

#### 情况 B：新的项目内文档上下文

**触发条件：** 用户提到了项目内的文档路径

**示例：**
```
用户："请你根据项目内的文档 docs/example.md，实现项目内的功能 B"
```

**判断逻辑：** 检查 `.agents/project-docs-index.md` 中是否已存在该文档

**如果不存在：** 向用户提问：
> 是否要将这个新的文档添加到 `.agents/project-docs-index.md` 索引文件中？

**注意：**
> 你需要明确用户提到的某个目录是文档相关的目录，还是需求实现相关的目录。如果是文档相关的目录，添加到项目文档索引文件中；如果是需求实现相关的目录，不要询问用户是否将该目录添加到索引文件中。

#### 情况 C：新的编码约束

**触发条件：** 用户提到了编码规范、约束或规则

**示例：**
```
用户："请你根据 xxx 这个约束，实现项目内的功能 C"
用户："使用xxx来开发需求"
用户："使用xxx打包构建项目"
用户："我们需要遵循 Airbnb 的 JavaScript 编码规范"
用户："项目必须使用 TypeScript 严格模式"
用户："所有 API 接口都需要添加统一的错误处理"
用户："组件开发必须遵循原子设计原则"
用户："数据库操作必须使用事务来保证数据一致性"
用户："不要使用xxx，而要使用xxx"
用户："不要将xxx直接xxx，应该xxx"（实现过程中发现的约束）
```

**判断逻辑：** 检查 `.agents/constraints.md` 中是否已存在该约束相关内容

**注意：** 除了用户明确提到的约束外，在**实现过程中发现/创建的约束**也应该被检测。例如：
- 实现了某个功能后发现需要遵循的规范（如"SSE data 必须是 JSON"）
- 修复 bug 后总结的注意事项（如"不要将 Part 类型直接字符串化"）

**如果不存在：** 向用户提问：
> 是否要将这个新的约束添加到约束目录中？

### 步骤 3：更新索引文件

当用户确认需要添加新上下文时，执行更新：

#### 更新远程文档索引

在 `.agents/remote-docs-index.md` 中添加新条目，包含：
- 文档 URL
- 文档描述（从用户提示中提取或询问用户）
- 获取方式（直接读取链接/使用专有的 cli 等方式）
- 添加日期

#### 更新项目文档索引

在 `.agents/project-docs-index.md` 中添加新条目，包含：
- 文档路径
- 文档描述
- 添加日期

#### 更新编码约束

检查约束主题是否存在：
- **如果主题不存在：** 在 `.agents/constraints.md` 中创建新主题，然后将约束添加到该主题下
- **如果主题存在：** 直接将新约束添加到对应主题中

### 步骤 4：确认更新

更新完成后，向用户确认：
> 已将新上下文添加到索引中。如果需要查看或修改，可以直接编辑对应的索引文件。

## 重要提示

- **不要主动添加**：只有在用户明确提到新上下文时，才询问是否需要添加，并且你不能直接修改相关的索引文档，务必询问用户确认。
- **增量更新**：直接修改现有索引文件，不要覆盖整个文件
- **上下文感知**：理解文档的实际用途，添加有意义的描述
- **尊重现有结构**：保持索引文件的格式一致性
- **主动识别**：在实现过程中，如果发现需要遵循的约束（如修复问题后总结的注意事项、实现后总结的规范），应该主动向用户确认是否需要添加到约束文件中
